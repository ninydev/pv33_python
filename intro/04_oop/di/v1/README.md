# Dependency Injection (DI) в Python

## Що таке DI?

**Впровадження залежностей (Dependency Injection, DI)** — це патерн проектування, при якому об'єкт не створює залежності, від яких він залежить, самостійно. Замість цього ці залежності "впроваджуються" в нього ззовні (зазвичай через конструктор).

Це дозволяє зробити код:
1.  **Слабко пов'язаним (Loosely coupled):** Класи не знають про конкретні реалізації один одного, лише про інтерфейси.
2.  **Легким для тестування:** Ви можете легко замінити реальну базу даних або сховище на "заглушку" (mock) під час тестів.
3.  **Гнучким:** Зміна способу збереження даних (наприклад, з локального диска на Azure Blob) не потребує зміни логіки самого сервісу.

## Простий метод реалізації (без сторонніх бібліотек)

У цьому прикладі ми реалізували DI за допомогою:
1.  **Інтерфейсу (`storage_interface.py`):** Визначає контракт (метод `save`).
2.  **Конкретних реалізацій:** `AzureBlobStorage`, `MinioStorage`, `LocalStorage`.
3.  **Сервісу (`avatar_service.py`):** Приймає будь-який об'єкт, що реалізує інтерфейс `StorageInterface`.
4.  **DI Контейнера (`di_container.py`):** Простий клас, який знає, яку саме реалізацію потрібно надати сервісу.

### Як це працює:

1.  Ми створюємо **інтерфейс** за допомогою модуля `abc` (Abstract Base Classes).
2.  Створюємо класи, що **успадковуються** від цього інтерфейсу.
3.  **Впроваджуємо** залежність через конструктор (`__init__`) класу `AvatarService`.
4.  Використовуємо **контейнер** для реєстрації та отримання потрібних об'єктів.

### Запуск прикладу:

Щоб побачити DI в дії, запустіть файл `main.py`:

```bash
python3 main.py
```

Ви побачите, як один і той самий сервіс працює з різними типами сховищ, не змінюючи свій внутрішній код.
